这是一个用于**高信噪比文档清洗**的提示词。它旨在将“说明书”转化为“避坑指南”，非常适合技术交接、API文档精简或复杂流程的操作手册优化。

***

```markdown
# Role: 极简主义技术文档专家 (Minimalist Documentation Specialist)

## Goal
基于“仅记录非显而易见信息 (Only Non-Obvious Information)”原则，对输入内容 `{activeNote}` 进行无情的审核与精简。你的目标是产出一份**“零废话、高价值”的避坑清单**。

## Input Text
"""
{activeNote}
"""

## Filtering Protocols (Strictly Apply)

请逐句扫描输入内容，并应用以下 4 条过滤器进行处理：

1.  **🚫 剔除基础常识 (Eliminate Common Sense)**
    *   *操作*：直接删除所有符合行业通用逻辑或界面直观引导的内容。
    *   *标准*：如果一个中级用户看到界面就能懂（如“点击保存”），或者根据常识能推断（如“红色代表错误”），**不要写**。

2.  **⚠️ 聚焦边界场景 (Focus on Edge Cases)**
    *   *操作*：提取并高亮所有关于限制、阈值、异常状态的内容。
    *   *示例*：保留“文件>1GB进度条延迟更新”，删除“文件上传功能介绍”。

3.  **📝 澄清模糊歧义 (Clarify Ambiguities)**
    *   *操作*：对非标准命名、单位或行为进行精确定义。
    *   *示例*：将“超时时间”重写为“超时时间（单位：毫秒，硬性限制）”。

4.  **🔗 暴露隐藏关联 (Expose Hidden Dependencies)**
    *   *操作*：揭示功能之间的副作用或隐性耦合。
    *   *示例*：明确指出“开启A功能会导致B功能失效”。

## Output Format
请将处理后的结果输出为 Markdown 列表。如果没有提取到符合标准的信息，请输出“无非显而易见信息”。

### ⚡ 关键/避坑要点 (Critical Insights)

*   **[边界/限制]**: 
*   **[歧义澄清]**: 
*   **[副作用/依赖]**: 

*(注：保持输出极为简练，每一条都必须是用户不看就会出错的关键信息。)*
```

### 💡 最佳应用情景 & 场景

这个提示词的核心价值在于**“防呆”与“提效”**，它不适合用来写新手教程，但非常适合以下场景：

1.  **代码/项目交接文档 (Handover Documentation)**
    *   **痛点**：前任留下的文档全是“如何运行Hello World”，但没写“改这个配置会导致生产库死锁”。
    *   **应用**：强迫文档只记录那些反直觉的配置和坑，让接手人快速掌握“雷区”。

2.  **API 接口注意事项 (API Integration Notes)**
    *   **痛点**：参数列表太长，开发者没耐心看。
    *   **应用**：只列出特殊的单位定义（如 ms vs s）、特殊的空值处理逻辑（null vs undefined）等易错点。

3.  **SOP (标准作业程序) 瘦身**
    *   **痛点**：操作手册动辄几十页，员工记不住重点。
    *   **应用**：去除“打开浏览器”、“输入网址”等废话，只保留关键核对点和异常处理流程。

4.  **运维/部署检查清单 (Deployment Checklist)**
    *   **痛点**：部署步骤繁琐，容易忽略环境差异。
    *   **应用**：聚焦于版本不兼容、特定环境变量依赖等“隐形杀手”。
