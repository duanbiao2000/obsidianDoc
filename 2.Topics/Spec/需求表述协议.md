# 需求表述协议：从情绪噪音到系统建模

## 1. 核心逻辑：需求即建模

正确表述需求不是“聊天”，而是**对现实问题的规格化建模**。

- **本质区别**：愿望/抱怨描述“感知”（快点、不好用）；需求描述“约束”（延迟 < 100ms、点击路径 < 3 步）。
- **核心公式**：需求 = 目标状态 + 输入/输出规格 + 边界约束。

## 2. SAFIR 需求五要素（Actionable）

| 要素                 | 说明      | 落地动作                |
| :----------------- | :------ | :------------------ |
| **S - Scope**      | 明确边界与范围 | 定义“做什么”和“绝对不做什么”。   |
| **A - Assumption** | 前提条件与依赖 | 标注数据质量、环境要求等外部变量。   |
| **F - Format**     | 输入输出的结构 | 确定接口协议、数据类型与 UI 交互。 |
| **I - Intent**     | 最终业务目的  | 解释“为什么要做”，防止手段替代目标。 |
| **R - Resolution** | 工程化执行层级 | 确保需求可被代码实现、可被测试验证。  |

## 3. 转化协议：从愿望到规格（Scenario）

**如果**你只有一个模糊愿望（如：“我想优化发帖功能”），**那么**请执行以下 3 步：

1. **量化指标**：将“优化”拆解为加载时间（≤1s）和成功率（≥99.9%）。
2. **定义边界**：明确支持的格式（JPG/PNG ≤ 10MB）与禁止项（单日上限 10 条）。
3. **锁定闭环**：定义异常流（如审核不通过时返回的具体 Code 和提示语）。

## 4. 需求表述水平审计 (Evaluation Matrix)

| 等级           | 表现特征       | 改进方向                  |
| :----------- | :--------- | :-------------------- |
| **L1 终端用户**  | 纯情绪、愿望、抱怨  | **引入规格**：增加数字和单位。     |
| **L2 初级 PM** | 流程罗列，缺逻辑闭环 | **寻找漏洞**：补齐异常分支与边缘情况。 |
| **L3 架构师**   | 结构化、有约束意识  | **解耦抽象**：定义模块接口与系统演进。 |
| **L4 顶级工程师** | 转化为可执行技术协议 | **极致精度**：消除任何自然语言歧义。  |
