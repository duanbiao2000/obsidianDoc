# [[编程范式]] | Minimal

---

## 0. 本质 (The Essence)
- **定义**：解决计算问题的**高层方法论**。
- **核心逻辑**：范式决定了你“如何思考问题”以及“如何组织逻辑”，而不仅仅是语法规则。
- **定位**：理解语言设计灵魂的“元认知”工具。

---

## 1. 核心范式矩阵

| 范式 | 核心逻辑 | 隐喻 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **面向过程** | 步骤清单 | 菜谱 / 流水线 | 底层开发、简单脚本 |
| **面向对象** | 对象交互 | 乐高积木 / 组织架构 | 大型复杂系统、GUI |
| **函数式** | 变换组合 | 数学函数 / 过滤器链 | 数据处理、高并发、状态敏感 |
| **声明式** | 描述结果 | 订单 / 装修蓝图 | SQL 查询、CSS 样式 |
| **逻辑式** | 规则推理 | 侦探破案 / 法律判决 | 人工智能、约束求解 |

---

## 2. 逻辑骨架 (Minimal Code)

- **面向过程**：`State = Initial; State = Step1(State); Step2(State);` (关注**如何做**)
- **面向对象**：`Object.Action(Data);` (关注**谁来做**)
- **函数式**：`Output = Compose(f, g, h)(Input);` (关注**变换什么**)
- **声明式**：`Goal(Condition);` (关注**想要什么**)

---

## 3. 避坑指南 (Brutal Truths)

- **范式不是牢笼**：现代语言（Python, JS, Java）多为**多范式**。强行在所有场景用单一范式是执念，而非专业。
- **OOP 的过度封装**：为了封装而封装会导致无意义的样板代码（Boilerplate），增加认知负担。
- **函数式的抽象代价**：函数式虽优雅，但在处理极其复杂的状态机或硬件底层时，性能和直观性可能不如命令式。
- **语法不等于范式**：用 Java 写出长达千行的 `main` 函数，本质依然是面向过程。

---

## 4. 决策树

1. **逻辑是简单的步骤流？**
    - → **面向过程**。
2. **系统包含复杂的实体及其关系？**
    - → **面向对象**。
3. **需要处理无状态的数据转换或高并发？**
    - → **函数式**。
4. **只需要描述最终状态或检索数据？**
    - → **声明式**。
5. **需要从已知事实推导未知结果？**
    - → **逻辑式**。

---

## 5. 跨领域迁移 (Thinking Model)

- **管理学**：
    - 面向过程 = SOP 标准作业程序。
    - 面向对象 = 部门职责定义与协作。
    - 函数式 = 扁平化的任务驱动流。

---

**原则**：工具决定思维。不要用锤子去拧螺丝，理解语言背后的范式，才能发挥其 100% 的效能。
