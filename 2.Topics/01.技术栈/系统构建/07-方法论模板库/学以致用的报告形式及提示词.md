# 学以致用：FOSS 产出执行协议

## 1. 核心模型

**产出价值：$Value = \frac{场景匹配 \times 逻辑闭环}{执行阻力}$**

| 报告类型           | 适用场景       | 核心防御/价值点          | 产出公式                     |
| :------------- | :--------- | :---------------- | :----------------------- |
| **技术调研 (TI)**  | 工具选型/深度对比  | 架构解耦、社区寿命、控制权     | TI = 功能对比 + 部署难度 + 风险评估  |
| **迁移指南 (MG)**  | 存量替换/流程重塑  | 数据无损、操作平替、SOP     | MG = 备份准备 + 转换步骤 + 故障排除  |
| **概念验证 (PoC)** | 落地实验/可行性测试 | 最小可行性 (MVP)、技术栈验证 | PoC = 部署指令 + 配置示例 + 潜力评估 |
| **成本效益 (CBA)** | 决策说服/管理层汇报 | 总拥有成本 (TCO)、供应商锁定 | CBA = (显性+隐性成本) vs 长期收益  |

## 2. 提示词执行协议 (SOP)

### A. TI 调研协议

- **指令**：`撰写一份关于 [工具] 的 TI 报告。`
- **必选项**：核心功能、技术架构、与 [商业软件] 对比、社区支持、推荐意见。
- **价值轴**：聚焦“自由”与“控制权”。

### B. MG 迁移协议

- **指令**：`编写从 [原件] 迁移至 [FOSS] 的 MG 指南。`
- **必选项**：备份清单、安装配置、文档转换逻辑、平替操作、FAQ。
- **原则**：步骤必须原子化、易跟随。

### C. PoC 验证协议

- **指令**：`设计 [工具] 的 PoC 项目报告。`
- **必选项**：项目目标、技术栈清单、部署 CMD/Yaml 示例、避坑记录、演示效果。
- **重点**：记录“遇到的挑战”及其解法。

### D. CBA 分析协议

- **指令**：`进行 [场景] 下商业 vs FOSS 的 CBA 分析。`
- **必选项**：TCO 比较（许可/托管/维护/培训）、效益项（数据主权/定制化潜力）。
- **终点**：给出明确的 Go/No-Go 结论。

## 3. 瞬时执行清单 (Checklist)

- [ ] **定位**：当前是要“对比”还是“落地”？（选 TI/CBA 或 MG/PoC）
- [ ] **变量**：目标软件和 FOSS 替代品是否已明确？
- [ ] **去噪**：是否排除了非核心功能的赘述？
- [ ] **闭环**：报告是否能直接触发下一步行动决策？
