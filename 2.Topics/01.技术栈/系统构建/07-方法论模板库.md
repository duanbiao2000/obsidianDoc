# 方法论文档库

> 本文档整合了产品化创作、报告编写和 FOSS 产出执行的相关方法论，为工程师提供系统化的内容创作能力。

---

## 📚 一、产品化创作框架

### 1. 四阶段框架

*源自：[[内容结构模式：产品化创作框架]]*

**核心逻辑**：内容即产品，将创作过程结构化为"产品发布"流程。

| 阶段 | 动作 | 目标 | 对应文档 | 产出物 |
|------|----------|----------|----------|
| **定义问题 (Problem)** | 识别技术债 | 遗留系统分析报告 | 技术债报告 |
| **发布方案 (Solution)** | 模块化更新 | 版本说明 (Release Notes) | v1.0 更新说明 |
| **执行文档 (Manual)** | 指令化引导 | 开发者手册 (Execution Guide) | 使用 SOP 文档 |
| **量化结果 (Results)** | 数据化汇报 | 性能看板 (Dashboard) | 性能对比报告 |

---

### 2. 核心代码架构

**逻辑流程**：
```python
# 1. 定义问题 (Problem)
debt = LegacySystem.analyze_tech_debt()

# 2. 发布方案 (Solution)
new_version = OptimizedProcessor(v2_0_features=["Parallel", "Cache"])

# 3. 执行文档 (Execution)
def deploy_workflow(data):
    check_env()      # 步骤1
    validate_data()  # 步骤2
    process_data()   # 步骤3
    return generate_report()

# 4. 量化结果 (Results)
roi = ROICalculator.compare(legacy=debt, new=current_metrics)
print(f"效率提升: {roi.efficiency_gain}%")
```

**关键原则**：
- 指令化：删除修饰语，将描述改为"指令"
- 版本化：为每一次方法论更新打上 Version Tag
- 闭环化：必须包含 ROI 评估，否则方案不完整

---

### 3. 选题矩阵：传统 vs. 产品发布

| 维度 | 传统模式 (Unboxing) | 产品发布模式 (Release) |
|------|----------|----------|----------|
| **视角** | 旁观者测试 | 开发者发布 |
| **逻辑** | 时间顺序 | 逻辑驱动 (Problem-Solution) |
| **受众** | 消费者 | 执行者 / 决策者 |
| **终点** | 感觉不错 | KPI 达标 |

**核心优化**：
- 指令化：将描述改为"指令"
- 版本化：为每一次方法论的更新打上 Version Tag
- 闭环化：必须包含 ROI 评估

## 📚 二、FOSS 产出执行协议

### 1. 核心模型

**产出价值公式**：
```
Value = (场景匹配 × 逻辑闭环) / 执行阻力
```

| 报告类型 | 核心防御/价值点 | 产出公式 |
|----------|----------|----------|----------|
| **技术调研 (TI)** | 架构解耦、社区寿命、控制权 | TI = 功能对比 + 部署难度 + 风险评估 |
| **迁移指南 (MG)** | 数据无损、操作平替、SOP | MG = 备份准备 + 转换步骤 + 故障排除 |
| **概念验证 (PoC)** | 最小可行性 (MVP)、技术栈验证 | PoC = 部署指令 + 配置示例 + 潜力评估 |
| **成本效益 (CBA)** | 总拥有成本 (TCO)、供应商锁定 | CBA = (显性+隐性成本) vs 长期收益 |

---

### 2. 提示词执行协议 (SOP)

#### A. TI 调研协议

**指令**：`撰写一份关于 [工具] 的 TI 报告。`

**必选项**：
- 核心功能
- 技术架构
- 与 [商业软件] 对比
- 社区支持
- 推荐意见

**价值轴**：聚焦"自由"与"控制权"

**输出格式**：
```markdown
# TI: [工具名称]

## 1. 技术概述
[技术介绍]

## 2. 架构对比
### 2.1 数据架构
| 特性 | [工具] | [商业软件] | 推荐 |
|------|----------|----------|----------|
| 分布式支持 | [X] | [Y] | 推荐使用分布式架构（可扩展性） |
| 数据一致性 | ACID 事务 | 最终一致性（可接受短暂不一致） | 推荐使用最终一致性（高可用成本） |
| 社区活跃度 | 活跃 | 不活跃 | 推荐活跃社区（长期支持） |
| 自托管成本 | 高 | 低 | 考虑云托管成本 |

### 2.2 部署运维
| 特性 | [工具] | [商业软件] | 推荐 |
|------|----------|----------|----------|
| 容器化支持 | Docker | 虚拟机镜像 | 推荐 Docker（标准化、生态成熟） |
| 监控能力 | 基础监控 | 集成监控平台 | 推荐开源监控方案（Prometheus + Grafana） |
| 自动化程度 | 手动部署 | CI/CD | 推荐 CI/CD 平台（降低运维负担） |
| 文档质量 | 简单 | 详细文档 | 推荐详细文档（减少沟通成本） |

## 3. 推荐意见
[基于上述分析，我们推荐使用 [工具]，理由如下：]

1. 架构设计符合团队技术栈
2. 社区活跃，长期支持有保障
3. 部署运维成本可控
4. 避免 vendor lock-in
```

#### B. MG 迁移协议

**指令**：`编写从 [原件] 迁移至 [FOSS] 的 MG 指南。`

**必选项**：
- 备份清单
- 安装配置
- 文档转换逻辑
- 平替操作
- FAQ

**原则**：步骤必须原子化、易跟随

**输出格式**：
```markdown
# MG: 从 [原件] 迁移至 [FOSS]

## 1. 前期准备
### 1.1 备份数据
- [ ] 用户数据（导出 CSV）
- [ ] 配置文件
- [ ] 自定义插件和主题

### 1.2 环境验证
- [ ] [FOSS] 版本兼容性测试
- [ ] 功能差异对比
- [ ] 性能基准测试

## 2. 迁移执行
### 2.1 安装 [FOSS]
```bash
# 包管理器安装
# [包管理器] install [FOSS名称]

# 配置迁移
# [FOSS] config import --export [原件配置]

# 数据迁移
# [FOSS] 数据导入工具 --import [备份文件]
```

### 2.2 平替操作
| 功能 | [原件] 操作 | [FOSS] 操作 |
|------|----------|----------|----------|
| 查询 | Ctrl+F | 搜索框 | 搜索框 |
| 编辑 | Ctrl+E | 编辑器 | 在线编辑器 |
| 报表 | Ctrl+R | 报表导出 | 数据透视表 |
| 导出 | Ctrl+S | 文件导出 | 文件导出 |

## 3. 常见问题 FAQ

**Q1: 数据迁移会丢失吗？**
**A**: 如果使用 [FOSS] 的数据导入工具，不会丢失。建议先备份原数据。

**Q2: 自定义配置如何迁移？**
**A**: 优先使用 [FOSS] 的配置文件格式，然后手动调整个性化配置。

**Q3: 环境适应期多久？**
**A**: 通常需要 1-2 周的日常使用适应。关键在于建立新的工作习惯，而不是追求 100% 功能一致。
```

## 1. 项目概述
- 目标：验证 [工具] 是否适合团队场景
- 范围：[具体功能范围]
- 技术栈：[列出所有技术]

git clone [FOSS仓库]

# 配置环境
export ENV=development
export DATABASE_URL=localhost:5432

# 启动服务
[FOSS启动命令] --port 8080
```

### 2.2 云部署
**服务**：[推荐云服务]
- Vercel：前端部署

## 3. 避坑记录
| 问题 | 现象 | 解决方案 | 预防措施 |
