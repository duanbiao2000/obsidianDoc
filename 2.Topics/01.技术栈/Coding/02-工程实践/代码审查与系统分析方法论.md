# [[代码审查与系统分析方法论]] | Minimal

---

## 0. 本质 (The Essence)

- **核心逻辑**：系统级思维 > 细节沉溺。代码只是实现的末节，系统架构才是灵魂。
- **阅读准则**：自顶向下 (Top-Down)，文档先行，结构为王。
- **定位**：从单纯的“读代码者”进化为“系统架构审计师”。

---

## 1. 核心七步法 (The 7-Step Workflow)

| 步骤         | 核心动作                        | 交付物        | 吐槽                  |
| :--------- | :-------------------------- | :--------- | :------------------ |
| **1. 文档**  | README $\rightarrow$ API 规范 | 全局背景       | 别直接跳进 `.py` 或 `.go` |
| **2. 结构**  | 目录树 + 模块边界分析                | 架构草图       | 搞清楚谁在统治谁            |
| **3. 流程**  | 追踪入口点 (Main/Handler)        | 序列图/调用链    | 跟着请求跑一遍             |
| **4. 工具**  | IDE 符号搜索 + 反向查找             | 高效跳转       | 手翻代码是原始人的活          |
| **5. 可视化** | 绘制状态机/依赖图                   | 脑图/Mermaid | 图表的理解效率是文字的10倍      |
| **6. 记录**  | 职责 + 陷阱 + 改进建议              | 审计笔记       | 不记笔记 = 没读           |
| **7. 测试**  | 研读单元/集成测试                   | 边界约束       | 测试用例是最好的文档          |

---

## 2. 逻辑骨架 (Minimal Hierarchy)

### **自顶向下抽象阶梯**

`Business (做什么?) -> Architecture (模块划分) -> Component (组件协作) -> Implementation (具体实现)`

### **流程优化闭环**

`映射 (Mapping) -> 找瓶颈 (Bottleneck) -> 定指标 (KPI) -> 迭代测试 (Iteration)`

---

## 3. 避坑指南 (Brutal Truths)

- **细节优先是死刑**：没搞清楚系统干啥就开始读函数实现，必陷入细节地狱。
- **无序阅读是浪费**：跳跃式阅读会导致认知负荷爆炸，必须按依赖顺序（底层 $\rightarrow$ 上层）推进。
- **代码味道 (Smell) 即信号**：长函数、硬编码、高耦合是系统崩溃的前兆，必须记录。
- **别信注释，信测试**：注释可能撒谎，但可运行的测试用例永远反映真实边界。

---

## 4. 诊断决策树

1. **“我不知道从哪开始”？**
   - $\rightarrow$ 强制执行 **步骤 1 & 2**：读 README，看目录树。
2. **“我陷入了实现细节”？**
   - $\rightarrow$ 立即**退后一步**。问：这个函数在全局流程中扮演什么角色？谁调用它？
3. **“系统运行太慢/不稳定”？**
   - $\rightarrow$ 执行**逆向分析法**。从目标反推依赖，识别关键路径上的失败点和约束。

---

## 5. 实施检查清单 (Final Checklist)

- [ ] **WHY**: 我知道系统存在的商业价值和核心目标吗？
- [ ] **ENTRY**: 我找到所有用户请求的入口点了吗？
- [ ] **DEPENDENCY**: 我画出核心模块之间的依赖方向了吗？（拒绝循环依赖）
- [ ] **RISK**: 我识别出单点故障、超卖风险或数据不一致的隐患了吗？
- [ ] **KPI**: 我定义了衡量优化成功与否的可量化标准（P99、吞吐量）吗？

---

**原则**：简单是最高级的复杂。先理清逻辑脉络，再修剪代码枝叶。
