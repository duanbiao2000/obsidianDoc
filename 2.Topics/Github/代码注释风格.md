### 一、 核心本质
**注释 = 意图传递**。
目标：弥补代码表达能力的不足，降低维护者的**认知开销**。

### 二、 注释选型矩阵

| 类型 | 核心目标 | 关注点 | 关键特征 |
| :--- | :--- | :--- | :--- |
| **文档型 (Doc)** | 接口标准化 | 参数、返回、异常 | 结构化、机器可解析 |
| **功能型 (Func)** | 解释设计意图 | “为什么”而非“做什么” | 聚焦逻辑，非语法搬运 |
| **结构型 (Struct)** | 视觉导航 | 模块划分、步骤引导 | 目录化、提升扫读效率 |
| **警示型 (Warn)** | 风险规避 | 技术债、修改约束、TODO | 包含警告、关联 PRD |
| **调试型 (Debug)** | 临时排查 | 变量状态、异常追踪 | 临时性、PR 前必删 |

### 三、 核心行动原则 (Principles)
- **拒绝冗余**：不解释代码显而易见的行为（如 `i++ // i 加 1`）。
- **代码说“什么”，注释说“为什么”**：代码描述逻辑，注释描述背景与决策。
- **关联业务**：复杂的业务规则需标注需求文档 ID (如 `Ref: PRD-056`)。
- **及时清理**：过期的注释比没注释更危险；Debug 注释严禁进入生产分支。
