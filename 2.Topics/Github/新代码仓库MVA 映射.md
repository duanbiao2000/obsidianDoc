## 一、分析瘫痪的真实来源（Repo 场景）

❗ 文件过多 → 认知负荷↑\
❗ 架构未知 → 不敢修改\
❗ 依赖复杂 → 担心跑不起来\
❗ 历史包袱 → 害怕理解错误

★ 本质\
→ 把「理解」当作行动前置条件

---

## 二、MVA 核心重构（学习 Repo）

★ 目标重定义\
❗ 不是“理解系统”\
★ 而是“制造一次有效接触”

---

## 三、Repo 学习 MVA 四步法

### ★ Step 1：强制缩小入口

★ 只回答 3 个问题

- Entry Point 在哪？（main / app / server）
- 一次请求/流程从哪进？
- 最先跑起来的命令是啥？

❗ 忽略：完整架构 / 所有模块

---

### ★ Step 2：最小运行闭环

★ 行动

- clone
- install deps
- run / build / test

❗ 能跑 ≻ 知道为什么能跑\
❗ 报错 ≻ 看文档

★ 报错即学习素材

---

### ★ Step 3：单点追踪法

★ 只选一个真实路径

- 一个 API
- 一个 CLI command
- 一个 UI 操作

★ 手段\
→ 打 log / debugger / print

❗ 不追全链路\
★ 只追“我碰到的这条线”

---

### ★ Step 4：最小改动验证

★ 做一件“无害但真实”的事

- 改一个 log
- 改一个 return 值
- 加一个注释 / test

❗ 能改 ≻ 懂设计\
★ 改动 = 心理 ownership 产生点

---

## 四、Repo 学习的反直觉原则

❗ 不要先画架构图\
❗ 不要先看所有 README\
❗ 不要按目录顺序读

★ 正确顺序\
→ 跑 → 断 → 改 → 回溯

---

## 五、MVA 带来的真实收益

★ 仓库从“外物” → “可操纵对象”\
★ 抽象恐惧 → 具体问题\
★ 文档从负担 → 工具\
★ 学习从被动 → 主动

---

## 六、自检清单（5 分钟）

- 今天我有没有 **run 过 repo**？
- 我是否 **跟踪过一条真实执行路径**？
- 我是否 **亲手制造过一次可回滚改动**？

❗ 如果都没有 → 我在分析瘫痪

---

## 七、元结论

★ Repo 不是用来“理解”的\
★ Repo 是用来“对抗”的

❗ Ownership 永远来自修改，不来自阅读
