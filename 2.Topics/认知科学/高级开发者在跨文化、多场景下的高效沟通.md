### 一、 核心本质
**目标**：降低沟通 RTT（往返耗时），实现“一次性信息对齐”。
**底层逻辑**：结构化信息 + 文化适配。

### 二、 文化适配矩阵
| 维度 | 美国风格 (Direct) | 中文环境 (Contextual) | **高级策略 (Hybrid)** |
| :--- | :--- | :--- | :--- |
| **结论** | 开门见山 | 铺垫后出 | **结论先行 + 逻辑支撑** |
| **冲突** | 直言不讳 | 委婉保护面子 | **分离事实与观点** |
| **决策** | 强势推荐 | 模糊开放 | **提供选项 + 明确倾向** |

### 三、 三大核心场景模版

#### 1. Code Review (4层递进)
- **直接表态**：一句话说明风险（如：有 Race Condition 风险）。
- **具体位置**：文件 + 行号 + 代码块。
- **影响评估**：为什么重要（如：生产环境 10K QPS 会崩溃）。
- **方案对比**：建议 A（推荐）、建议 B（备选）、建议 C（折中）+ 你的倾向。

#### 2. 生产 Bug 排查 (极速对齐)
- **状态**：🔴 严重性 (P1-P3) | 影响用户数 | 持续时间。
- **症状**：观察到的指标变化（Error Rate / Latency）。
- **诊断**：已排查项 (✅) | 疑似项 (❓) | 关联部署 (PR#)。
- **行动**：计划 A (Rollback) / B (Fix)，明确需要对方提供的信息。

#### 3. 系统设计 (分层解释)
- **产品/PM**：谈业务指标、转化率、用户体验。
- **工程师**：谈具体实现、逻辑流程、代码影响。
- **架构师**：谈 SLA、权衡 (Trade-offs)、扩展性。

### 四、 “零往返”检查清单 (One-shot Info)
- [ ] **明确结论**：邮件/消息开头即是目的。
- [ ] **数据支撑**：用表格/指标代替“我觉得”。
- [ ] **多选方案**：至少提供 2 个选项，避免“非黑即白”。
- [ ] **行动终点 (CTA)**：明确截止日期（Deadline）和具体决策人。

### 五、 极简金句
**沟通不是为了证明你是对的，而是为了让团队最快做出正确的决定。**
