## 核心观点

优秀的工程师应停止被动收藏各种“定律”，而应主动将其翻译成可执行的启发式算法，用以指导日常的技术选型、系统设计和团队协作决策。

## 关键要点

- **核心思想**：从定律收藏家转变为决策架构师。
- **林迪效应启发式**：核心业务用“老古董”，边缘创新用“新玩具”。
- **奥卡姆剃刀启发式**：先用 `if` 解决问题，再考虑过度抽象。
- **反脆弱性启发式**：主动注入故障，强制系统进化。
- **第一性原理启发式**：停止说“最佳实践”，开始问“为什么”。

## 详细内容

### 核心洞察：从收藏家到架构师

将抽象原则编译成大脑中可执行的决策子程序，是从博学的工程师到顶尖架构师的关键跃迁。关键在于应用，而非仅仅记忆。

### 支撑论据：定律到启发式的转化框架

#### 1. 林迪效应 (Lindy Effect)

- **启发式**：“核心业务用‘老古董’，边缘创新用‘新玩具’。”
- **应用场景**：技术选型时，为数据库、消息队列等关键组件选择经受过长期考验的技术（如 PostgreSQL），以此规避新技术的隐性风险。

#### 2. 奥卡姆剃刀 (Occam's Razor)

- **启发式**：“先用 `if`，再想多态。”
- **应用场景**：开发新功能时，用最直接的方式实现。拒绝任何在需求明确前就进行的过早抽象，只在模式重复出现时才进行重构。

#### 3. 反脆弱性 (Antifragility)

- **启发式**：“主动注入故障，强制系统进化。”
- **应用场景**：在系统设计评审中，必须回答：“如果这个依赖挂了，系统会优雅降级还是直接崩溃？” 定期通过混沌工程来验证和强化系统的韧性。

#### 4. 复杂性守恒 (Complexity Conservation)

- **启发式**：“简化是有代价的，代价由谁承担？”
- **应用场景**：设计一个极简的 API 意味着复杂性被转移到了后端。必须评估这种转移是否增加了维护成本和不可控的风险。

#### 5. 第一性原理 (First Principles)

- **启发式**：“停止说‘最佳实践’，开始问‘为什么’。”
- **应用场景**：在重构或架构决策时，回归问题的本质。例如，不要问“我们该用什么微服务框架？”，而是问“我们要解决的根本问题是组织扩张快，还是业务逻辑解耦？”

#### 6. 汉隆的剃刀 (Hanlon's Razor)

- **启发式**：“优先假设是愚蠢，而不是恶意。”
- **应用场景**：排查跨团队 Bug 时，首先反思“我们的 API 文档是否有歧义？”，而不是指责“他们又写了烂代码！”。这能将冲突转为协作。
