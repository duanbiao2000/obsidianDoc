# 标签使用指南

本文档说明如何在Obsidian知识库中使用三层标签体系。

---

## 标签体系概述

本知识库采用**三层结构化标签系统**，用于组织和管理所有笔记。标签是知识图谱的主要组织结构，优先级高于文件夹分类。

### 三层标签结构

```
#Domain/<SubDomain>     # 领域/子领域 - 表示知识所属的领域
#Status/<State>         # 状态 - 表示笔记的当前状态
#Type/<ContentType>     # 类型 - 表示笔记的内容类型
```

---

## 第一层：Domain（领域）

### 作用
定义笔记所属的知识领域或学科范畴，用于纵向组织知识。

### 预定义的Domain

| Domain | 中文名称 | 适用范围 | 示例子标签 |
|--------|---------|---------|-----------|
| `#Domain/Protocol` | 协议与规范 | 流程、规范、最佳实践 | `#Domain/Protocol/Work`, `#Domain/Protocol/Governance` |
| `#Domain/TechStack` | 技术栈 | 编程、工具、技术 | `#Domain/TechStack/Programming`, `#Domain/TechStack/DevOps` |
| `#Domain/Cognition` | 认知系统 | 思维模型、学习方法 | `#Domain/Cognition/MentalModel`, `#Domain/Cognition/Method` |
| `#Domain/Content` | 内容创作 | 写作、表达、创作技巧 | `#Domain/Content/Writing`, `#Domain/Content/Editing` |
| `#Domain/Career` | 职业发展 | 管理、领导力、职业规划 | `#Domain/Career/Management`, `#Domain/Career/Planning` |
| `#Domain/Health` | 生活与健康 | 健康、运动、休息 | `#Domain/Health/Fitness`, `#Domain/Health/Wellness` |
| `#Domain/Language` | 语言与移民 | 语言学习、移民规划 | `#Domain/Language/Learning`, `#Domain/Language/Immigration` |

### Domain选择原则

1. **优先选择预定义Domain** - 尽量使用已存在的Domain标签
2. **单Domain原则** - 每个笔记通常只属于一个主要Domain
3. **可创建子Domain** - 如现有Domain不够，可在Domain下创建子标签（如 `#Domain/Cognition/StudyMethod`）
4. **跨Domain笔记** - 如果笔记涉及多个领域，选择最主要的领域，其他领域通过[[双向链接]]关联

### 示例

```yaml
# 技术学习笔记
tags: [#Domain/TechStack/Programming]

# 认知模型笔记
tags: [#Domain/Cognition/MentalModel]

# 职业管理笔记
tags: [#Domain/Career/Management]
```

## 第二层：Status（状态）

### 作用
表示笔记的当前状态，用于跟踪内容生命周期。

### 预定义的Status

| Status | 含义 | 使用场景 |
|--------|------|---------|
| `#Status/TODO` | 待办 | 需要完善、补充或后续处理的笔记 |
| `#Status/Review` | 审查中 | 需要定期回顾、更新的笔记 |
| `#Status/Done` | 已完成 | 内容完善、稳定不变的笔记 |
| `#Status/Archive` | 已归档 | 历史内容，仅供参考 |

### Status使用原则

1. **默认从TODO开始** - 新建笔记时标记为 `#Status/TODO`
2. **定期更新Status** - 根据笔记完善程度调整状态
3. **避免多Status** - 一个笔记在同一时间只能有一个Status
4. **Status与Type无关** - 无论笔记类型如何，都可以有Status标签

### 示例

```yaml
# 新建的学习笔记
tags: [#Status/TODO]

# 定期回顾的笔记
tags: [#Status/Review]

# 完善的参考资料
tags: [#Status/Done]
```

## 第三层：Type（类型）

### 作用
表示笔记的内容形式或结构类型，用于区分不同类型的笔记。

### 预定义的Type

| Type | 含义 | 使用场景 | 建议结构 |
|------|------|---------|---------|
| `#Type/Note` | 原子笔记 | 单个概念、想法的精简笔记 | Zettelkasten模板 |
| `#Type/Reference` | 参考资料 | 外部资源、引用、摘录 | 来源链接 + 摘要 |
| `#Type/Journal` | 日志记录 | 每日记录、反思、心流 | 日期 + 关键事件 |
| `#Type/Keynote` | 演讲笔记 | 演讲、会议、课程笔记 | 要点总结 + 行动项 |
| `#Type/Clipping` | 网页剪藏 | 从外部收集的原始内容 | 原始内容 + 标注 |
| `#Type/Meta` | 元数据笔记 | 关于系统本身、方法论 | 系统文档、方法论 |
| `#Type/Project` | 项目笔记 | 项目规划、执行记录 | 项目模板 |
| `#Type/Template` | 模板笔记 | 可复用的模板 | 模板结构 |

### Type使用原则

1. **单Type原则** - 每个笔记属于一个主要类型
2. **Type与目录结构对应** - Type应与笔记的用途和结构匹配
3. **优先使用模板** - 使用 `5.Misc/Template/` 下的对应模板创建笔记
4. **可混合使用** - 某些笔记可以同时有Type和Domain/Status

### 示例

```yaml
# 原子笔记
tags: [#Type/Note]

# 参考资料笔记
tags: [#Type/Reference]

# 演讲笔记
tags: [#Type/Keynote]
```

## 完整标签示例

### 示例1：技术学习笔记

```yaml
---
tags:
  - Domain/TechStack/Programming
  - Status/Review
  - Type/Note
  - programming
  - javascript
---
```

**解释**：
- `#Domain/TechStack/Programming` - 属于技术栈/编程领域
- `#Status/Review` - 需要定期回顾（技术知识需要持续学习）
- `#Type/Note` - 是一个原子笔记（单个概念）
- `#programming`, `#javascript` - 具体技术标签（可选）

---

### 示例2：每日日志

```yaml
---
tags:
  - Domain/Health/Wellness
  - Status/Done
  - Type/Journal
  - daily
---
```

**解释**：
- `#Domain/Health/Wellness` - 健康/福祉领域
- `#Status/Done` - 已完成（日志无需修改）
- `#Type/Journal` - 日志类型
- `#daily` - 日常标记

---

### 示例3：项目笔记

```yaml
---
tags:
  - Domain/Career/Management
  - Status/TODO
  - Type/Project
  - project-management
---
```

**解释**：
- `#Domain/Career/Management` - 职业/管理领域
- `#Status/TODO` - 进行中（项目需要跟踪）
- `#Type/Project` - 项目类型
- `#project-management` - 项目管理相关

---

### 示例4：参考资料笔记

```yaml
---
tags:
  - Domain/Cognition/MentalModel
  - Status/Done
  - Type/Reference
  - mental-model
  - first-principles
---
```

**解释**：
- `#Domain/Cognition/MentalModel` - 认知/思维模型领域
- `#Status/Done` - 已完成（参考资料一般不变）
- `#Type/Reference` - 参考资料类型
- `#mental-model`, `#first-principles` - 具体概念标签

## 标签压缩规则

为简化标签使用，以下标签可以压缩为规范的三层标签：

| 原标签 | 替代标签 |
|--------|---------|
| `#todo` | `#Status/TODO` |
| `#review` | `#Status/Review` |
| `#done` | `#Status/Done` |
| `#archive` | `#Status/Archive` |
| `#note` | `#Type/Note` |
| `#reference` | `#Type/Reference` |
| `#journal` | `#Type/Journal` |
| `#keynote` | `#Type/Keynote` |
| `#clipping` | `#Type/Clipping` |
| `#meta` | `#Type/Meta` |
| `#project` | `#Type/Project` |
| `#template` | `#Type/Template` |

**注意**：压缩规则适用于迁移和整理，新建笔记时直接使用三层标签。

## 常见问题（FAQ）

### Q1: 如何选择合适的Domain？

**A**:
1. 先看预定义的Domain表，找到最匹配的领域
2. 如果不确定，选择涵盖范围更广的Domain（如 `#Domain/Cognition` vs `#Domain/Cognition/MentalModel`）
3. 查看已有笔记的Domain选择，保持一致性
4. 当笔记涉及多个领域时，选择最主要的领域，其他通过[[双向链接]]关联

---

### Q2: Status应该何时更新？

**A**:
- `#Status/TODO` → `#Status/Review`: 笔记内容基本完善，但需要定期回顾和更新
- `#Status/Review` → `#Status/Done`: 笔记已稳定，不需要频繁修改
- `#Status/TODO` → `#Status/Archive`: 笔记不再相关，但保留历史价值
- **建议**: 每周回顾时检查Status状态

---

### Q3: 笔记可以有多个Domain标签吗？

**A**:
- **推荐**: 单Domain原则，通过[[双向链接]]关联其他领域
- **例外**: 某些跨领域笔记可以有2个Domain标签，但应避免超过3个
- **原则**: 选择1个主Domain + 通过链接关联其他相关笔记

---

### Q4: 标签和文件夹是什么关系？

**A**:
- **标签**是主要组织结构，用于知识图谱和发现
- **文件夹**是辅助组织结构，用于PARA方法（Projects/Topics/Resources/Archives）
- **优先级**: 标签 > 文件夹
- **实践**: 同一文件夹下的笔记可能有不同的Domain标签

---

### Q5: 如何处理标签重复或冲突？

**A**:
1. **标签格式错误**: 如 `#Domain/TechStack` 和 `#domain/techstack`，统一为小写且用斜杠分隔
2. **语义重复**: 如 `#productivity` 和 `#Domain/Productivity`，合并为 `#Domain/Productivity`
3. **命名冲突**: 如 `#note` 和 `#Type/Note`，使用规范的 `#Type/Note`
4. **定期清理**: 使用Obsidian的Tag Pane查看所有标签，删除或合并重复标签

---

### Q6: 创建新标签需要注意什么？

**A**:
1. **优先使用预定义标签** - 查看本文档和 `Atlas/Index/仓库标签管理系统.md`
2. **保持命名一致性** - 使用英文、驼峰命名或斜杠分隔（如 `#Domain/TechStack`）
3. **避免过细分类** - 标签层级不超过3层（如 `#Domain/TechStack/Programming` 而非 `#Domain/TechStack/Programming/JavaScript`）
4. **记录新标签** - 如果创建了新的Domain，更新 `Atlas/Index/仓库标签管理系统.md`

---

### Q7: 标签数量有建议吗？

**A**:
- **最少**: 每个笔记至少3个标签（1个Domain + 1个Status + 1个Type）
- **建议**: 每个笔记3-6个标签（包括具体概念标签）
- **上限**: 避免超过8个标签（过多的标签降低可维护性）

---

### Q8: 如何通过标签快速找到相关笔记？

**A**:
1. 使用Obsidian的Tag Pane按标签筛选笔记
2. 使用Dataview插件查询特定Domain/Status/Type的笔记
3. 使用[[双向链接]]跳转到相关笔记
4. 查看目录下的 `_Index_of_<DirectoryName>.md` 索引文件

## 最佳实践

### 1. 新建笔记时的标签策略

```markdown
步骤1: 确定笔记类型
  - 如果是概念笔记 → #Type/Note
  - 如果是参考资料 → #Type/Reference
  - 如果是每日记录 → #Type/Journal

步骤2: 选择Domain
  - 查看预定义Domain表，选择最匹配的领域

步骤3: 设置初始Status
  - 新建笔记默认为 #Status/TODO
  - 如果是外部参考资料，可直接用 #Status/Done

步骤4: 添加具体标签（可选）
  - 添加与内容直接相关的具体概念标签
  - 避免过于通用的标签（如 #useful, #important）
```

### 3. 保持标签一致性

**检查清单**:
- [ ] 所有标签使用统一格式（小写 + 斜杠分隔）
- [ ] 压缩标签使用三层规范格式（如 `#todo` → `#Status/TODO`）
- [ ] Domain标签匹配预定义的Domain表
- [ ] 每个笔记至少有Domain + Status + Type三个标签
- [ ] 标签总数不超过8个

---

### 4. 利用标签提升知识发现

**策略**:
1. **按Domain探索** - 查看 `#Domain/Cognition` 下的所有笔记，发现知识关联
2. **按Status过滤** - 查看 `#Status/Review` 下的笔记，定期回顾
3. **按Type查询** - 查看 `#Type/Reference` 下的笔记，快速找到参考资料
4. **结合使用** - 查询 `#Domain/TechStack + #Type/Note` 下的所有技术笔记
