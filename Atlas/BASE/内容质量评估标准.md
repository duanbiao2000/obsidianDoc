## 🧠 一句话总纲（Executive Summary）

> **知识库的核心价值不在于笔记数量，而在于每篇笔记的"可复用性"和"可发现性"。**
>
> 本标准旨在建立一套**客观、可量化、可持续改进**的内容质量评估体系，确保所有笔记达到"高可用性"标准。

---

## 📊 质量评估四维度框架

基于 [claude.md范例.md](4.Archives/Tools/claude.md范例.md) 的质量维度定义，我们建立了以下评估框架：

### 1️⃣ 清晰度（Clarity）- 权重 30%

**定义**: 笔记是否能够被快速理解，是否存在歧义

#### 评估标准（0-5 分制）

| 分数 | 标准 | 具体要求 |
|-----|------|---------|
| **5 分 - 优秀** | 完全无歧义 | • 核心术语有明确定义<br>• 逻辑流程清晰，无跳跃<br>• 使用精确的语言，避免模糊表达<br>• 有示例或类比辅助理解 |
| **4 分 - 良好** | 基本清晰 | • 核心概念明确<br>• 逻辑连贯<br>• 偶有不够精确的表达，但不影响理解 |
| **3 分 - 一般** | 勉强可懂 | • 核心概念存在，但定义不够明确<br>• 逻辑有跳跃<br>• 需要读者自行推理部分内容 |
| **2 分 - 较差** | 理解困难 | • 核心概念模糊或缺失<br>• 逻辑混乱<br>• 存在歧义表达，需要多次阅读 |
| **1 分 - 不合格** | 无法理解 | • 核心概念未定义<br>• 逻辑断裂<br>• 表达混乱，完全无法理解 |

#### 检查清单

- [ ] 核心术语在首次出现时是否有明确定义？
- [ ] 是否避免了使用"这个"、"那个"等模糊指代？
- [ ] 逻辑流程是否清晰，是否有明确的因果关系？
- [ ] 是否提供了具体示例来说明抽象概念？
- [ ] 是否避免了过度使用专业术语（除非在特定技术栈中）？

---

### 2️⃣ 结构（Structure）- 权重 25%

**定义**: 笔记的组织结构是否合理，是否易于导航和检索

#### 评估标准（0-5 分制）

| 分数 | 标准 | 具体要求 |
|-----|------|---------|
| **5 分 - 优秀** | 结构完整 | • 有明确的章节层次（H1-H3）<br>• 包含目录/摘要/总结<br>• 内部链接≥3 个，指向相关笔记<br>• 有适当的列表、表格、图表 |
| **4 分 - 良好** | 结构清晰 | • 章节层次分明<br>• 有内部链接（2-3 个）<br>• 使用列表和格式化增强可读性 |
| **3 分 - 一般** | 结构基本合理 | • 有基本的章节划分<br>• 内部链接≤2 个<br>• 格式化较少，可读性一般 |
| **2 分 - 较差** | 结构混乱 | • 章节划分不清或缺失<br>• 内部链接≤1 个或没有<br>• 缺乏基本的格式化 |
| **1 分 - 不合格** | 无结构 | • 没有章节划分<br>• 没有内部链接<br>• 大段文字堆砌 |

#### 检查清单

- [ ] 是否有明确的章节划分（至少 3 个二级标题）？
- [ ] 是否包含了"一句话总纲"或摘要？
- [ ] 是否有内部链接指向相关笔记（至少 3 个）？
- [ ] 是否使用了列表、表格等格式化工具？
- [ ] 是否有关键词高亮或引用块？
- [ ] 长笔记（>500 字）是否有目录或总结？

---

### 3️⃣ 深度（Depth）- 权重 25%

**定义**: 笔记是否提供了足够的背景上下文和深度思考

#### 评估标准（0-5 分制）

| 分数 | 标准 | 具体要求 |
|-----|------|---------|
| **5 分 - 优秀** | 深度充分 | • 提供背景上下文（为什么这个概念重要）<br>• 包含决策理由或权衡分析<br>• 有实际案例或数据支持<br>• 有个人思考或反思 |
| **4 分 - 良好** | 深度适中 | • 有基本背景说明<br>• 有一定推理过程<br>• 有示例支持 |
| **3 分 - 一般** | 深度不足 | • 背景说明简略<br>• 主要是概念罗列，缺少推理<br>• 缺少示例 |
| **2 分 - 较差** | 深度很浅 | • 无背景说明<br>• 只有定义，无推理<br>• 无示例 |
| **1 分 - 不合格** | 无深度 | • 只有零散信息<br>• 无上下文<br>• 无推理过程 |

#### 检查清单

- [ ] 是否说明了"为什么"这个概念重要？
- [ ] 是否提供了"怎么做"的具体步骤或方法？
- [ ] 是否包含了决策理由或权衡分析？
- [ ] 是否有实际案例、数据或引用支持观点？
- [ ] 是否有个人思考、反思或未来展望？
- [ ] 技术笔记是否包含了"踩坑记录"或"最佳实践"？

---

### 4️⃣ 可维护性（Maintainability）- 权重 20%

**定义**: 笔记是否易于更新、维护和避免信息冗余

#### 评估标准（0-5 分制）

| 分数 | 标准 | 具体要求 |
|-----|------|---------|
| **5 分 - 优秀** | 高可维护性 | • YAML frontmatter 完整（tags, update, rating）<br>• 单一真实源（无重复内容）<br>• 有明确的更新机制或过期标记<br>• 引用外部资源有链接 |
| **4 分 - 良好** | 可维护 | • YAML 基本完整<br>• 基本无重复内容<br>• 有 update 字段 |
| **3 分 - 一般** | 勉强可维护 | • YAML 不完整（缺少部分字段）<br>• 存在少量重复内容<br>• update 字段过期 |
| **2 分 - 较差** | 难以维护 | • YAML 严重缺失<br>• 有明显的内容重复<br>• 无 update 字段或长期未更新 |
| **1 分 - 不合格** | 不可维护 | • 无 YAML frontmatter<br>• 大量重复内容<br>• 无任何版本信息 |

#### 检查清单

- [ ] YAML frontmatter 是否包含必需字段（tags, update, rating）？
- [ ] tags 是否遵循三层标签系统（Domain/Status/Type）？
- [ ] 是否与其他笔记存在重复内容？
- [ ] update 字段是否为最近日期（3 个月内）？
- [ ] 是否引用了外部资源（书籍、文章、视频）？
- [ ] 是否使用了 Obsidian 的嵌入功能 `![[` 避免重复粘贴？

## 📈 综合质量评分计算

### 总分计算公式

```
总分 = 清晰度×0.3 + 结构×0.25 + 深度×0.25 + 可维护性×0.2
```

**分数范围**: 0-5 分

### 质量等级划分

| 等级 | 分数范围 | 标签 | 行动建议 |
|-----|---------|------|---------|
| 🌟 **优秀** | 4.5-5.0 | `#Quality/Excellent` | • 可作为范例展示<br>• 放入"最佳实践案例库" |
| ✅ **良好** | 4.0-4.4 | `#Quality/Good` | • 符合标准，无需改进<br>• 定期检查（每季度） |
| 👍 **合格** | 3.0-3.9 | `#Quality/Acceptable` | • 可用但有小问题<br>• 标记待改进<br>• 1 个月内优化 |
| ⚠️ **需改进** | 2.0-2.9 | `#Quality/NeedsImprovement` | • 需要较大改进<br>• 2 周内完成优化<br>• 优先处理高访问量笔记 |
| ❌ **不合格** | 0-1.9 | `#Quality/Poor` | • 需要完全重写<br>• 1 周内完成<br>• 或移至 `4.Archives/Outdates/` |

## 🔍 质量评估工作流

### 新笔记创建时（自检）

1. **创建笔记**: 使用 [Zettelkasten模板.md](5.Misc/Template/Zettelkasten模板.md)
2. **填写内容**: 按照模板结构填写
3. **自检**: 使用下方"快速检查清单"
4. **打分**: 在 YAML 中添加 `rating` 字段
5. **打标签**: 添加质量标签 `#Quality/<等级>`

### 定期质量审计（每月）

1. **随机抽查**: 使用 Dataview 每月随机抽取 10% 笔记
2. **批量评估**: 使用 [质量评估模板.md](5.Misc/Template/质量评估模板.md)
3. **记录结果**: 更新 YAML `rating` 字段
4. **低质量处理**: 对 rating < 3 的笔记标记并改进
5. **进度跟踪**: 在质量看板中记录改进进度

## ✅ 快速检查清单（Quick Checklist）

新笔记创建完成后，使用此清单快速检查（< 2 分钟）：

### 基础要求（必须全部满足）

- [ ] **标题清晰**: 标题能准确概括笔记内容
- [ ] **核心概念定义**: 首次出现的重要术语有定义
- [ ] **章节划分**: 至少有 3 个二级标题
- [ ] **内部链接**: 至少 3 个 `[[wikilink]]`
- [ ] **YAML 完整**: 包含 tags, created, update, rating
- [ ] **标签规范**: 遵循三层标签系统

### 质量提升（建议满足）

- [ ] **一句话总纲**: 开头有摘要或核心观点
- [ ] **示例或案例**: 有具体例子说明抽象概念
- [ ] **反思或思考**: 有个人见解或未来展望
- [ ] **外部引用**: 有书籍、文章或视频链接
- [ ] **格式化**: 使用列表、表格、引用块等
- [ ] **总结或关键要点**: 结尾有总结或要点

### 可选加分项

- [ ] **图表或可视化**: 使用 Excalidraw、Mermaid 等
- [ ] **代码块**: 包含可运行的代码示例
- [ ] **任务清单**: 使用 Tasks 插件的 TODO 列表
- [ ] **Dataview 查询**: 包含自动化查询

## 📋 不同类型笔记的特殊标准

### 📝 原子笔记（Zettelkasten）

**核心要求**: 单一概念,高度可链接

| 维度 | 权重 | 特殊要求 |
|-----|------|---------|
| 清晰度 | 35% | • 核心概念必须在首句定义<br>• 避免包含多个不相关概念 |
| 结构 | 20% | • 简洁结构（<500 字）<br>• 内部链接≥3 个 |
| 深度 | 25% | • 提供背景上下文<br>• 有个人思考 |
| 可维护性 | 20% | • YAML 必须完整<br>• 避免与其他笔记重复 |

### 🗂️ 项目笔记（Projects）

**核心要求**: 目标导向,可跟踪

| 维度 | 权重 | 特殊要求 |
|-----|------|---------|
| 清晰度 | 30% | • 项目目标明确<br>• 交付物清晰 |
| 结构 | 25% | • 必须有时间线或里程碑<br>• 有 TODO 列表 |

**核心要求**: 信息准确,可追溯

| 维度 | 权重 | 特殊要求 |
|-----|------|---------|
| 清晰度 | 25% | • 信息准确无误<br>• 避免主观臆断 |
| 结构 | 30% | • 有清晰的索引或目录<br>• 分类合理 |
| 深度 | 20% | • 提供来源或引用<br>• 有背景说明 |

### 🎓 学习笔记（Topics）

**核心要求**: 知识体系化,可复习

| 维度 | 权重 | 特殊要求 |
|-----|------|---------|

| 结构 | 25% | • 知识点层次分明<br>• 有思维导图或概念图 |
| 深度 | 25% | • 有理解过程<br>• 有疑问或待解决问题 |
| 可维护性 | 20% | • 定期复习标记<br>• 与其他知识点链接 |

## 🎯 质量改进优先级策略

当发现大量低质量笔记时,按以下优先级处理：

### 优先级 1: 高访问量 + 低质量

**选择标准**:
- `view-count` > 10 且 `rating` < 3
- 或位于 `1.Projects/` 目录且 `rating` < 3

**行动**:
- 立即改进（1 周内完成）
- 改进后重新评分
- 记录改进日志

### 优先级 2: 技术栈 + 低质量

**选择标准**:
- 位于 `2.Topics/01.技术栈/` 且 `rating` < 3

**行动**:
- 2 周内完成改进
- 重点补充"踩坑记录"和"最佳实践"

### 优先级 3: 近期创建 + 低质量

**选择标准**:
- `created` < 6 个月且 `rating` < 3

**行动**:
- 1 个月内完成改进
- 避免积累技术债务

### 优先级 4: 其他低质量笔记

**行动**:
- 分批次处理（每季度 10%）
- 或移至 `4.Archives/Outdates/`

## 📊 质量监控指标

### 全局指标（Dataview 查询）

```dataview
TABLE
  round(average(float(rating)), 2) AS "平均质量分",
  length(rows) AS "笔记总数",
  length(rows[rating >= 4]) AS "优秀笔记数",
  length(rows[rating < 3]) AS "需改进笔记数"
FROM "1.Projects" OR "2.Topics" OR "3.Resources"
WHERE rating
GROUP BY "全库"
```

### 分类指标（按目录）

```dataview
TABLE
  round(average(float(rating)), 2) AS "平均质量分",
  length(rows) AS "笔记总数"
FROM "1.Projects" OR "2.Topics" OR "3.Resources"
WHERE rating
GROUP BY file.folder
```

### 时间趋势（按创建时间）

```dataview
TABLE
  round(average(float(rating)), 2) AS "平均质量分",
  length(rows) AS "笔记总数"
FROM "1.Projects" OR "2.Topics" OR "3.Resources"
WHERE rating
GROUP BY dateformat(date(created), "yyyy-MM")
```

## 🔄 持续改进机制

### 每月任务（自动化）

1. **质量审计**: 随机抽查 10% 笔记
2. **低质量处理**: 改进 rating < 3 的笔记
3. **质量报告**: 生成质量分布报告
4. **最佳笔记**: 评选"每月最佳笔记"

### 每季度任务

1. **全面评估**: 对所有笔记进行质量评分
2. **归档清理**: 将过时内容移至 Archives
3. **标准更新**: 根据实际使用情况更新本标准
4. **案例更新**: 更新"最佳实践案例库"

### 每年任务

1. **系统回顾**: 评估质量管理体系有效性
2. **标准修订**: 重大版本更新
3. **经验总结**: 形成年度质量报告
