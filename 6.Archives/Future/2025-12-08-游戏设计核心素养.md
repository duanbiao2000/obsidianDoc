# 游戏设计三层架构

## 核心认知：游戏设计是分层的系统

```
玩家体验 (What player feels)
         ↑
    三层架构
         ↓
商业可持续性 (What company earns)

层级：
  L3. 产品策略  ← 为谁设计？如何赚钱？如何竞争？
  L2. 系统设计  ← 所有子系统如何互动？如何平衡长期留存？
  L1. 玩法机制  ← 单个动作好不好玩？手感对吗？
```

---

## 第一层：Mechanics（玩法机制）

### 本质

```
问题：这个玩法单独有意思吗？
范围：单个跳跃、射击、收集
核心：打磨"手感"
```

### 6 大核心素养

|素养|具体表现|反面|
|---|---|---|
|**玩法原子理解**|知道哪些基础单元组合会产生有趣的 emergent gameplay|组合无意义，只是堆砌功能|
|**手感与反馈**|对输入延迟敏感；视觉反馈、音效反馈精心设计|"feel"生硬，没有爽感|
|**难度曲线**|让新手不挫折，老玩家有挑战|前期无聊，后期受挫|
|**约束条件创意**|通过限制创造策略空间（如移动速度↓，策略↑）|自由度高但无法则，游戏变无聊|
|**快速原型**|用纸张、粗糙代码验证玩法假设|全力开发后才发现玩法无趣|
|**玩家心理模型**|预测玩家会如何理解和滥用机制|被玩家找到 BUG 利用|

### 实例分析：《Celeste》冲刺跳跃

```
表面：跳跃 + 冲刺 = 简单机制

深层设计：
  ✓ 精细的输入反馈（立即响应，清晰的按键手感）
  ✓ 可预测的物理（非"魔法系"，用户能精确计算）
  ✓ 逐层递进的挑战（新手关卡只需基础操作，后期需要精妙组合）
  ✓ 约束条件创意（冲刺次数有限 → 必须精确规划路线）

结果：
  简单的机制 + 精细的手感 + 递进的难度
  = 极高的满足感 + 玩家愿意反复尝试
```

## 第二层：Systems（系统设计）

### 本质

```
问题：所有系统加起来，玩家的长期体验是什么？
范围：经济、进度、社交、等级 等多个子系统
核心：让系统互相支撑，避免失衡
```

### 6 大核心素养

|素养|具体表现|反面|
|---|---|---|
|**系统建模**|把经济、进度、社交看作互联的有机体；理解反馈环|各系统独立设计，产生冲突|
|**参数平衡**|掌握指数/线性/对数增长的应用；用数据模型预测|参数调整靠直觉，经常失衡|
|**长期留存**|设计足够的内容更新、成就、社交激励|前期爽，后期枯燥，玩家流失|
|**防止滥用**|预见破坏性用法（OP Build、快速 farming）并设计护栏|被玩家找到漏洞，经济崩盘|
|**数据驱动**|读懂 cohort analysis、retention curve、monetization funnel|靠感觉改动数值，改坏了不知道为什么|
|**跨系统协调**|玩法改动 → 经济影响？新社交功能 → 时间分配如何变化？|改了一个地方，另一个地方冒泡|

### 关键数据概念速查

```
DAU (Daily Active Users) = 每天活跃人数
MAU (Monthly Active Users) = 每月活跃人数
Churn Rate = 流失率（每周离开的人数百分比）
Cohort = 同批玩家（比较"第 1 天玩家"vs"第 30 天玩家"的行为差异）
LTV (Lifetime Value) = 单个玩家一生的贡献度
Retention Curve = 玩家留存曲线（通常是"悬崖跌"形状）

系统设计的目标 = 让 Retention Curve 尽量平缓（而不是急剧下降）
```

### 实例分析：《暗黑破坏神 3》的平衡调整

```
问题：某职业 Build 过度强大，破坏经济平衡
      物品掉落率过低，玩家无法感受进度
      难度系数导致新手受挫

系统级解决：
  ✓ 调整职业技能数值（不是简单削弱，而是改变策略空间）
  ✓ 调整物品掉落率（同时调整难度，维持挑战感）
  ✓ 重新设计难度曲线（平衡新手-老玩家体验）

特点：
  单个改动涉及多个系统
  需要预测全局影响
  通常需要多次迭代才能达成平衡
```

## 第三层：Strategy（产品策略）

### 本质

```
问题：为谁设计？如何赚钱？如何保持竞争力？
范围：商业模式、玩家获取、版本计划、社区管理
核心：把游戏做成可持续的产品
```

### 8 大核心素养

|素养|具体表现|反面|
|---|---|---|
|**用户洞察**|不只是"谁玩"，而是"为什么选这个而非竞品"|假设用户需求，被市场教育|
|**商业模式**|一次性/内购/订阅的权衡|变现和体验冲突，玩家反感|
|**LTV 优化**|理解 DAU/MAU、留存率、ARPU 的关系|增长快但留存烂，最后钱赚不回来|
|**竞争定位**|找到差异化切口（蓝海思维）|红海竞争，无法脱颖而出|
|**心理学定价**|理解为什么某个价格点吸引用户|定价随意，转化率低|
|**版本策略**|何时发布？针对谁？如何保持热度？|内容枯竭，玩家快速流失|
|**社区与营销**|让玩家成为传播者；减少社区毒性|玩家自发流失，口碑变差|
|**A/B 测试**|用实验验证假设（UI、价格、新玩法）|靠直觉改动，经常改错方向|

### 商业模式的本质区别

```
一次性购买（买断制）：
  收入来源：初始销售
  挑战：用户买完就走，难以维持长期收入
  适合：单机、故事线游戏

免费 + 内购：
  收入来源：小额多次购买（cosmetic、battle pass、加速）
  挑战：需要精心设计，别让付费影响体验
  适合：多人、对战、生存类游戏

订阅制：
  收入来源：月度/年度固定收入
  挑战：内容更新压力大，用户预期高
  适合：服务型游戏、MMO
```

### 实例分析：《堡垒之夜》的产品策略

```
不是单纯的游戏，而是"文化现象"

关键决策：
  ✓ 跨界合作（Marvel、Travis Scott 演唱会）
    → 把游戏变成"社交场景"，不只是竞技
  
  ✓ Battle Pass 的季度更新
    → 创造持续的目标感和购买欲望
  
  ✓ Live Event 的社交性
    → 让玩家一起见证重大事件，增强社区感
  
  ✓ 免费入场 + 高端皮肤定价
    → 降低进入门槛，提高鲸鱼变现

结果：
  不是"最好玩的游戏"，而是"最文化相关的游戏"
  = 持续的商业可持续性
```

## 第四部分：三层的关系矩阵

### 改动在三层的传导

```
【Mechanics 层改动】
  "增加新的冲刺能力"
        ↓
  【Systems 层影响】
  → 职业平衡改变
  → 关卡难度需要调整
  → 玩家的时间投入改变
        ↓
  【Strategy 层影响】
  → 新玩家学习曲线变化
  → 老玩家新鲜感
  → 可能影响付费转化

一个看似简单的改动，实际是"三层震荡"
```

### 反例：不协调的三层

```
Mechanics 设计得很爽
但 Systems 里存在 OP Build
→ 玩家用 Build 秒杀所有内容
→ 失去挑战感

Systems 设计得很平衡
但 Strategy 的付费模式太激进
→ 玩家觉得"付费即胜利"
→ 竞技感被破坏

Strategy 很清晰（赚钱目标明确）
但 Mechanics 手感生硬
→ 玩家体验差
→ 再好的商业模式也救不了
```

## 第五部分：快速判断法

### 看一个游戏的设计水平

```
看 Mechanics：
  问："这个游戏单个动作有意思吗？"
  好的游戏：即使不看故事，单纯操作都很舒服
  差的游戏：必须靠故事或视觉效果撑起来

看 Systems：
  问："长期玩会不会感到失衡或无聊？"
  好的游戏：进度、挑战、奖励的反馈环设计精妙
  差的游戏：前期爽，后期要么太难要么太容易

看 Strategy：
  问："为什么我玩这个而不是竞品？"
  好的游戏：有清晰的差异化价值（社交、创新、品质）
  差的游戏：靠价格低或免费硬撑，无法维持热度

三层都好 = 现象级游戏（如《原神》《艾尔登法环》）
两层好一层差 = 有瑕疵但可玩的游戏
一层好两层差 = 快要死的游戏
```

## 第六部分：核心素养的本质

### 为什么这些素养重要？

```
Mechanics 素养：
  本质 = 对"人的感受"的敏感度
  成本 = 反复打磨、大量玩家测试
  
Systems 素养：
  本质 = 系统思维（看全局，不只看单点）
  成本 = 学习复杂的数学模型和数据分析
  
Strategy 素养：
  本质 = 商业直觉（理解用户价值、竞争格局）
  成本 = 大量市场观察和A/B测试
```

### 最常见的误区

```
❌ 误区 1：Mechanics 爽就够了
  现实：短期吸引，长期无法维持

❌ 误区 2：系统设计就是"调数值"
  现实：调数值是最后一步，前面要做系统建模

❌ 误区 3：有好的商业模式就能成功
  现实：没有好玩的 Mechanics，商业模式也救不了

✅ 现实：三层都必须打磨，缺一不可
```

## 第七部分：从工程师角度的启发

### 如果你用 Python 设计游戏系统

```python
# Mechanics 层（输入 → 反馈）
class Player:
    def jump(self, input_strength):
        # 精细反馈设计：输入→视觉→音效→物理
        return self.calculate_trajectory(input_strength)

# Systems 层（多个子系统的交互）
class GameSystem:
    def __init__(self):
        self.economy = EconomySystem()
        self.progression = ProgressionSystem()
        self.social = SocialSystem()
    
    def update(self):
        # 子系统互相影响（重点是反馈环）
        self.economy.update()
        self.progression.update()  # 影响经济
        self.social.update()  # 影响留存

# Strategy 层（数据驱动的决策）
class ProductStrategy:
    def analyze_cohorts(self):
        # 分析不同批次玩家的行为差异
        d1_retention = self.get_retention(day=1)
        d7_retention = self.get_retention(day=7)
        # 如果 d7 << d1，说明 Systems 层有问题
    
    def ab_test(self, feature_a, feature_b):
        # 用实验验证假设，不靠直觉
        results_a = self.test(feature_a)
        results_b = self.test(feature_b)
        return self.analyze_statistical_significance(results_a, results_b)
```

**核心：** 工程师的系统思维 = 游戏设计的三层架构

```
模块化（Mechanics）
        ↓
系统交互（Systems）
        ↓
数据优化（Strategy）
```

## 最后：你应该问的问题

### 评估一个游戏设计

```
L1. Mechanics 层
  "操作有意思吗？手感对吗？"
  是 → 继续
  否 → 基础有问题，很难救

L2. Systems 层
  "长期不会感到失衡或无聊吗？"
  是 → 继续
  否 → 需要重新平衡所有系统

L3. Strategy 层
  "有清晰的用户价值和商业可持续性吗？"
  是 → 可以成功
  否 → 好游戏也可能失败

三个都是"是" = 极大概率成功 ✅
有一个"否" = 可能成功（靠补救）
两个"否" = 需要大改
```

### 改进自己的设计

```
如果只是美术设计师：
  → 理解 Mechanics（手感 = 视觉反馈）

如果是系统设计师（如你的 Python 背景）：
  → 深入 Systems 和 Strategy（数据模型、优化指标）

如果想做制作人：
  → 三层都要理解，特别是 Strategy
```

## 资源速查

### 理解三层的最小集合

```
Mechanics：
  看：《Celeste》gameplay、《超级马里奥 64》camera 设计
  学："Designing Games" by Tynan Sylvester

Systems：
  看：任何 GaaS 游戏的平衡补丁说明
  学："Game Balance and Mechanics" 和数据分析

Strategy：
  看：游戏公司的财报（Activision、EA）、商业分析
  学：A/B testing methodology、cohort analysis
```

### 你可以立即做的事

```
1. 选一个你经常玩的游戏
   分别用"三层框架"评估它
   → 你会发现它为什么成功或失败

2. 看一个失败的游戏（如某个转向手游的大作）
   分析：哪一层设计得不好？
   → 理解"三层不协调"的具体影响

3. 如果你用 Python 做游戏或系统设计
   → 用"Systems 层"的思维重构你的代码架构
   → 会发现代码的模块化对应游戏的系统平衡
```
