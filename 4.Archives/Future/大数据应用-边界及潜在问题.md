# 大數據應用：規模與效率的權衡指南

## 1. 核心邏輯：空間換時間 vs. 精度換速度

大數據處理的本質是解決**規模與效率的根本矛盾**。當傳統計算失效時，必須依賴專門數據結構與概率算法。

## 2. 核心數據結構矩陣 (Toolbox)

| 結構               | 核心機制       | ROI 洞察 (Actionable)                       |
| :--------------- | :--------- | :---------------------------------------- |
| **Heap (堆)**     | 動態維護極值     | **TopK 問題神器**。海量數據取前 N 名，無需全量排序，顯著降低 I/O。 |
| **Trie (字典樹)**   | 前綴路徑共享     | **海量文本檢索**。處理字符串前綴匹配與自動補全，空間利用率極高。        |
| **Bloom Filter** | 位向量 + 哈希映射 | **高效過濾器**。判斷“一定不存在”或“可能存在”，以極小誤差換取海量存儲節省。 |
| **HashMap**      | 鍵值對映射      | **O(1) 快速查找**。大規模數據快速定位與去重的底層基石。          |

## 3. 處理原則 (Processing Principles)

- **分而治之 (Divide & Conquer)**：數據分片，將大問題拆解為並行的子任務，適配分佈式計算。
- **內存高效表示**：使用位圖 (Bitmap) 或壓縮存儲，最大化單機處理能力的上限。
- **流式處理 (Streaming)**：針對持續產出的數據流進行實時分析，而非等待全量入庫後批處理。
- **概率算法**：在可控誤差範圍內，用概率性判斷（如 Bloom Filter）對抗指數級增長的數據規模。

## 4. 決策指南 (Scenario-Based)

- **場景 A：攔截緩存穿透/無效請求**
  - *策略*：在 DB 前部署 **Bloom Filter**，快速過濾不存在的 Key，保護底層存儲。
- **場景 B：從億級日誌提取搜索熱詞**
  - *策略*：**HashMap** 統計頻率 + **Heap (小頂堆)** 維護 TopN。
- **場景 C：海量網址過濾/黑名單**
  - *策略*：使用 **Trie** 樹進行前綴快速攔截或 **Bloom Filter** 進行初步判定。
