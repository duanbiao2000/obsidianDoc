# Netflix vs Google vs Amazon vs Meta

## 技术组织模型对照（Engineer 视角）

> **核心差异一句话版**

- Netflix：**用人才密度替代流程**
- Google：**用系统与流程放大中上水平工程师**
- Amazon：**用机制与约束逼出可规模化执行**
- Meta：**用速度与市场反馈筛选正确方向**

---

## 一、组织设计的“第一性目标”不同

| 公司                                  | 第一性目标      | 组织设计隐含假设     |
| ----------------------------------- | ---------- | ------------ |
| **Netflix**                         | 决策质量 > 一致性 | 人均判断力极高      |
| **Google**                          | 正确性 & 稳定性  | 大量聪明但背景不一致的人 |
| **Amazon**                          | 可规模化执行     | 人会犯错，系统要兜底   |
| **Meta**                            | 市场速度 & 试错  | 错是常态，慢是致命的   |
| 👉 所有后续差异，都是这个起点的推论。                |            |              |
| ---                                 |            |              |
| ## 二、人才模型：你是“放大器”还是“可替换部件”          |            |              |
| ### 1️⃣ Netflix：**Engineer = 决策节点** |            |              |

- 招聘门槛极高
- 默认你：
  - 能独立权衡 trade-offs
  - 理解 business impact
  - 为错误负责

## 📌 **你是系统的一部分，不是执行单元**

### 2️⃣ Google：**Engineer = 高质量组件**

- 招的是 top X%，但规模巨大
- 用：
  - Design Doc
  - Code Review
  - SRE / Error Budget
  - 强规范 infra

## 📌 **系统负责兜底，个人负责局部最优**

### 3️⃣ Amazon：**Engineer = 可预测执行单元**

- Bar Raiser 保证下限
- 但你被假设：
  - 会犯错
  - 会短视
  - 会 optimize local metrics

## 📌 **机制先行，trust 后置**

### 4️⃣ Meta：**Engineer = 快速实验者**

- 对个人能力容忍度相对高
- 用：
  - 快速合并
  - 快速上线
  - 快速 rollback

## 📌 **方向靠市场校准，不靠内部共识**

## 三、流程设计：谁在“背锅”？

| 公司        | 流程密度 | 出问题时谁背锅     |
| --------- | ---- | ----------- |
| Netflix   | 极低   | 个人          |
| Google    | 高    | 系统 + Review |
| Amazon    | 极高   | Mechanism   |
| Meta      | 中    | 市场 / 数据     |
| **残酷现实**： |      |             |

> 自由越多，个人风险越大。

---

## 四、技术决策机制（这是工程师最痛的地方）

### Netflix

- 没有 RFC 文化
- 强调：
  - context sharing
  - individual judgment

## 📌 优点：快\
📌 代价：你不能“躲在流程后面”

### Google

- Design Doc 是核心基础设施
- 目标：
  - 防止聪明人做蠢事
  - 降低 bus factor

## 📌 优点：稳\
📌 代价：慢、paperwork 重

### Amazon

- 6-page narrative
- 强制逻辑链条
- 强制 trade-off 显性化

## 📌 优点：执行力极强\
📌 代价：精神消耗大（writing tax）

### Meta

- Decision often = Ship → Measure
- 设计文档次要
- 数据说话

## 📌 优点：探索效率极高\
📌 代价：技术债积累快

## 五、并发视角（System Thinker Bonus）

把它们当成并发系统：

| 公司      | 并发控制策略                 |
| ------- | ---------------------- |
| Netflix | 乐观并发（Optimistic）       |
| Google  | 悲观并发（Locks everywhere） |
| Amazon  | 事务型系统（ACID over org）   |
| Meta    | Eventually Consistent  |

> Netflix 的 bug 是 catastrophic\
> Google 的 bug 是 expensive\
> Amazon 的 bug 是 exhausting\
> Meta 的 bug 是 noisy

## 六、对工程师的现实选择指南（很重要）

### 你适合 Netflix，如果你：

- 不需要明确边界
- 能承受“我就是 owner”
- 享受高认知密度对话
- 接受 **no safety net**

### 你适合 Google，如果你：

- 喜欢系统性正确
- 擅长写 doc、review
- 在复杂 infra 中成长
- 想长期复利

### 你适合 Amazon，如果你：

- 执行力怪兽
- 不怕压力与指标
- 接受强约束换成长
- 想快速晋升（or burn）

### 你适合 Meta，如果你：

- 对速度上瘾
- 容忍混乱
- 喜欢把用户当评审
- 能忍技术债

---

## 七、前瞻性判断（2025+）

- **Netflix 模型**：\
  → 在 AI / 高杠杆 IC 时代更有优势\
  → 但 hiring 是最大瓶颈
- **Google 模型**：\
  → 在 infra / research 仍然无敌\
  → 在速度战中吃亏
- **Amazon 模型**：\
  → 在规模化 AI 产品上仍然强\
  → 人才流失是长期隐患
- **Meta 模型**：\
  → 在 consumer AI / 新形态产品上最激进\
  → 稳定性靠时间换
