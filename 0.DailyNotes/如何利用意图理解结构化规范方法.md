# 如何利用"意图理解→结构化规范"方法

---

## 一、核心洞察

这句话揭示了一个关键的 AI 使用原则：

```
模糊需求 → 意图解析 → 结构化规范 → 高质量输出

大多数人的问题：
直接把模糊需求丢给 AI → 得到模糊的输出 → 反复修改 → 浪费时间

正确的做法：
先完成"意图解析"这一步，再让 AI 执行
```

**你可以自己扮演"意图理解智能体"，或者让 AI 帮你完成这一步。**

---

## 二、三种实践方法

### **方法 1：自己先结构化，再交给 AI**

```
❌ 糟糕的提示：
"帮我写一个用户登录功能"

✅ 结构化后的提示：
"实现用户登录功能，规范如下：

【功能需求】
- 支持邮箱+密码登录
- 支持 OAuth（Google、GitHub）
- 登录失败 5 次后锁定 15 分钟

【技术约束】
- 后端：FastAPI + PostgreSQL
- 认证：JWT，有效期 24 小时
- 密码：bcrypt 哈希

【安全要求】
- 防止暴力破解
- 防止时序攻击
- 日志记录所有登录尝试

【输出格式】
- 提供完整代码
- 包含单元测试
- 附带 API 文档"
```

---

### **方法 2：让 AI 帮你完成意图解析**

当你自己不确定需求时，**先让 AI 当"意图理解智能体"**：

```
第一步提示：
"我想做一个在线预约系统。
请帮我把这个需求解析为结构化规范，包括：
1. 核心功能列表
2. 用户角色和权限
3. 关键数据实体
4. 需要我澄清的问题"

AI 输出：
【核心功能】
- 用户注册/登录
- 查看可用时段
- 创建/取消预约
- 预约提醒通知
...

【需要澄清】
- 预约的是什么？（医生？会议室？课程？）
- 是否需要支付功能？
- 预约冲突如何处理？

第二步：回答 AI 的问题，确认规范

第三步：基于确认后的规范，让 AI 开始实现
```

---

### **方法 3：使用固定模板强制结构化**

创建一个"需求规范模板"，每次使用 AI 前先填写：

```markdown
## 需求规范模板

### 1. 目标陈述（一句话）
[我想要实现什么？]

### 2. 背景与约束
- 技术栈：
- 现有系统：
- 时间/资源限制：

### 3. 功能需求（按优先级）
- P0（必须有）：
- P1（应该有）：
- P2（可以有）：

### 4. 非功能需求
- 性能：
- 安全：
- 可维护性：

### 5. 成功标准
- 怎样算"做完了"：
- 怎样算"做好了"：

### 6. 已知风险/不确定性
- [列出你不确定的地方]
```

## 三、实际对比示例

### 场景：让 AI 帮你写一个爬虫

**未结构化版本：**
```
"帮我写个爬虫抓取新闻网站"
```

AI 可能的困惑：
- 哪个网站？
- 抓什么内容？
- 存到哪里？
- 要不要处理反爬？
- 用什么库？

**结构化规范版本：**
```
"实现一个新闻爬虫，规范如下：

【目标网站】
- 36氪（https://36kr.com）
- 只抓取科技频道

【抓取内容】
- 文章标题
- 发布时间
- 作者
- 正文（纯文本）
- 原文链接

【技术规范】
- 语言：Python 3.11
- 库：requests + BeautifulSoup（不用 Selenium）
- 存储：SQLite 本地数据库
- 并发：单线程，请求间隔 2 秒

【反爬处理】
- 随机 User-Agent
- 失败重试 3 次

【输出要求】
- 模块化代码结构
- 包含使用说明
- 错误处理完善"
```

## 四、进阶技巧：迭代式意图澄清

```
┌─────────────────────────────────────────────────────┐
│  Round 1: 粗略描述                                   │
│  "我想做一个任务管理工具"                            │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Round 2: AI 提问澄清                                │
│  "请回答以下问题：                                   │
│   - 个人用还是团队用？                               │
│   - 需要什么视图？（列表/看板/日历）                  │
│   - 要不要提醒功能？                                 │
│   - Web/移动端/桌面？"                               │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Round 3: 用户回答 + AI 生成结构化规范               │
│  "基于你的回答，需求规范如下：..."                   │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Round 4: 确认规范 → 开始实现                        │
└─────────────────────────────────────────────────────┘
```

**触发 AI 进入"意图解析模式"的提示词：**
```
"在开始实现之前，请先：
1. 列出你对需求的理解
2. 指出可能有歧义的地方
3. 提出 3-5 个澄清问题
4. 等我确认后再继续"
```

## 五、与前序知识的关联

这个方法本质上是 **Plan 思维在 AI 使用中的应用**：

| Plan 原则 | 在 AI 使用中的对应 |
|-----------|-------------------|
| 前期思考降低后期风险 | 先结构化需求，减少返工 |
| 强制慢思考 | 填写模板强制你想清楚 |
| 暴露理解差异 | AI 提问帮你发现自己没想到的 |
| 可反驳性 | 规范中明确"什么情况下换方案" |

---

## 六、一句话总结

> **不要让 AI 猜你想要什么——先把需求变成规范，再让 AI 执行规范。**

这相当于你在使用 AI 时，**自己扮演"意图理解智能体"的角色**，或者**让 AI 先帮你完成这一步**。

## 多智能体架构优势

在竞争激烈的智能体编程领域，DeepCode的差异化优势在于其精密的多智能体架构。与单一模型方法不同，DeepCode协调一套专业化智能体：

- **中央协调智能体**：主导工作流执行并协调任务分配
==- **意图理解智能体**：将用户需求解析为结构化规范==
- **文档解析智能体**：解读技术文档和研究论文
- **代码规划与参考挖掘智能体**：分析技术栈并搜索代码库
- **代码生成智能体**：将工作流输出合成为可执行代码
