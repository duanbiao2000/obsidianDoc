在软件开发和工程协作的语境中，**RFC** 是 **“Request for Comments”**（征求意见稿）的缩写。

---

### 一、RFC 的含义

- **起源**：RFC 最早由互联网工程任务组（IETF）在 1969 年引入，用于记录和标准化网络协议（如 TCP/IP、HTTP 等）。这些文档公开发布，邀请社区评审和反馈。
- **现代用法**：在许多技术公司（如 Google、Meta、Netflix、GitHub 等）和开源项目中，**RFC 被用作一种轻量级的设计文档机制**，用于在投入大量开发资源前，对重大技术决策、架构变更或新功能进行**提案、讨论和共识达成**。

---

### 二、RFC 与 Issue 的关系

|对比项|GitHub Issue|RFC|
|---|---|---|
|**目的**|跟踪 bug、任务、功能请求等|提出并讨论**技术设计方案**|
|**形式**|通常较简短，以问题描述为主|结构化文档（背景、目标、方案、权衡、替代方案等）|
|**粒度**|可大可小（从“修复拼写错误”到“重构整个模块”）|通常用于**中大型、有争议或影响广泛**的变更|
|**位置**|通常在 GitHub/GitLab 的 Issues 或 Discussions 中|可能是：  <br>• 一个特殊的 Issue（标题带 [RFC]）  <br>• 一个 Markdown 文件（如 `docs/rfc/001-auth-design.md`）  <br>• 专用平台（如 Notion、Google Docs）|
|**生命周期**|创建 → 讨论 → 关闭/完成|提案 → 评审 → 接受/拒绝 → 归档|

> ✅ 很多团队会用 **Issue 来承载 RFC**，例如创建一个标题为 `[RFC] 新用户认证系统设计` 的 Issue，在评论区或正文里贴出完整设计文档，供大家讨论。

---

### 三、RFC 典型结构（简化版）

```markdown
# [RFC] 引入 GraphQL API

## 背景
当前 REST API 难以满足前端灵活查询需求...

## 目标
- 减少前端请求数量
- 提升数据获取效率

## 提议方案
采用 Apollo Server + GraphQL Schema...

## 替代方案
1. 优化现有 REST（但无法解决 N+1 问题）
2. 使用 gRPC（学习成本高）

## 潜在风险
- 运维复杂度上升
- 缓存策略需重新设计

## 决策
待团队评审后决定是否推进。
```

---

### 四、何时使用 RFC？

建议在以下情况发起 RFC：

- 引入新技术栈（如从 MySQL 切换到 PostgreSQL）
- 重大架构调整（微服务拆分、事件驱动改造）
- 影响多个团队的接口变更
- 存在多种可行方案，需要集体决策

---

### 总结

> **RFC = Request for Comments**  
> 它不是 GitHub 的原生功能，而是一种**工程实践**，常借助 Issue、文档或 Wiki 来实现。  
> 相比普通 Issue，RFC 更强调**前瞻性设计、方案权衡和团队共识**，是成熟工程文化的重要体现。

如果你看到 “在 Issue 或 RFC 中讨论”，意思通常是：  
👉 **简单问题用 Issue，复杂/重大设计用 RFC（可能以 Issue 形式存在）**。
