# [[2025-12-11-AI在写代码领域的当前水平]]

## 1. 核心逻辑：能力边界矩阵 (Capability Boundary Matrix)

**系统定位：** AI = 高速 AST 生成器 + 不知疲倦的初级实习生。

| 等级 | 任务范畴 | 自主度 | 准确率预期 | 瓶颈因 (Bottleneck) |
| :--- | :--- | :--- | :--- | :--- |
| **Lv 2** | 单元测试、样板代码、正则 | **全自动** | 90%+ | 语法语义对齐。 |
| **Lv 3** | 跨模块重构、性能优化 | **辅助** | 60-70% | 全局上下文丢失；隐性依赖。 |
| **Lv 4** | 系统架构、技术选型 | **仅建议** | 30-40% | 缺乏工程约束与成本意识。 |
| **Lv 5** | 需求澄清、复杂权衡、旧账治理 | **无能** | <10% | 涉及社会工程与非结构化决策。 |

## 2. 决策协议：独立开发触发器 (Independence Algorithm)

**执行路径 (Execution Path)：**
1. **输入熵检 (Input Entropy)**：需求是否具备 $O(1)$ 的清晰度？（测试用例/接口定义）。
2. **复杂度约束 (Complexity)**：代码是否 $\leq 50$ 行且单一职责？
3. **风险评估 (Risk)**：是否涉及生产环境或敏感数据？

- **✅ 独立完成**：内部工具、原型代码、单元测试。
- **⚠️ 必须评审**：生产逻辑、并发控制、资源管理、安全接口。

## 3. 故障模式 (System Failure Modes)

| 模式 | 表现 (Symptom) | 探测协议 (Detection) |
| :--- | :--- | :--- |
| **假设钙化** | 缺乏 Null/异常检查。 | 静态扫描 + 单元测试。 |
| **并发坍塌** | 竞态条件 (Race Condition)。 | 压力测试 / 并发注入。 |
| **资源泄漏** | 打开 File/Conn 未关闭。 | 长期运行内存/句柄监测。 |
| **业务漂移** | 逻辑自洽但偏离需求。 | 人工逻辑审计 (Logic Audit)。 |
| **依赖腐败** | 使用废弃 API 或不兼容版本。 | 编译链路检查。 |

## 4. 验证协议 (Validation Protocol)

### **安全审查 Checklist**
- [ ] **注入防御**：SQL 参数化、XSS 转义。
- [ ] **数据隐私**：敏感字段加密、日志脱敏。
- [ ] **权限校验**：身份验证锚点检查。

### **质量审查 Checklist**
- [ ] **边界测试**：空值、溢出、零值处理。
- [ ] **资源一致性**：`defer`/`try-with-resources` 闭环。
- [ ] **可读性**：命名语义化，拒绝 AI 风格的冗长注释。

## 5. 执行指南：生产力放大协议 (Scaling Protocol)

### **P1: TDD 驱动 (Test-Driven AI)**
- **Step 1**：人工定义测试用例（断言集）。
- **Step 2**：AI 实现函数，直至测试通过。
- **价值**：建立硬性成功指标。

### **P2: 架构锁定 (Architecture First)**
- **Step 1**：人工设计接口契约、Repository 与 Service 层级 [^4]。
- **Step 2**：AI 填充原子方法。
- **价值**：防止 AI 引导的架构熵增。

### **P3: 多轮迭代 (Iterative Refinement)**
- **Loop 1**：MVP 逻辑生成。
- **Loop 2**：注入异常处理与日志。
- **Loop 3**：安全与性能 Hack。
