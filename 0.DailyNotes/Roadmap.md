如果项目中没有 `TODO` 或 `FIXME` 注释，说明该项目可能处于：**1. 极度成熟（很少见）**；**2. 快速迭代中忽略了注释**；或 **3. 处于“飞机稿”向“生产级”转化的真空期**。

在这种情况下，不能靠“提取”，而要靠 AI 的**“推断”与“对标”**。作为 Google 工程师，你可以通过以下三个维度的脚本化分析，喂给 AI Agent 来生成精准的 Roadmap：

---

### 维度一：工程缺陷分析 (基于 [[2025-12-10-高质量开源项目的特征]])

AI 需要知道项目“缺什么”，而不是“有什么”。

**1. 自动化扫描清单：**
让 AI 检查项目结构的完整性：
```bash
# 检查是否缺失关键工程化文件
ls -R | rg "tests|docs|workflows|docker|Makefile|pyproject\.toml"
```

**2. 给 AI 的 Prompt 策略：**
> **Prompt:**
> 我将提供该项目的目录结构和 `README.md`。
> 请对标 **Production-Ready** 的开源项目标准（如：完善的测试覆盖、CI/CD 自动化、容器化支持、文档网站），列出该项目在**工程化维度**上缺失的组件。
> **任务：** 将这些缺失项转化为 Roadmap 的 P0/P1 任务。

---

### 维度二：核心链路与复杂度分析 (Hotspot Analysis)

通过分析代码的“厚度”和“依赖密度”，让 AI 识别出哪些模块需要重构或扩展。

**1. 提取素材：**
```bash
# 1. 识别最大的几个模块（通常是逻辑密集区）
wc -l $(rg --files -t py) | sort -n | tail -10

# 2. 识别核心依赖（看出项目的技术边界）
rg -h "^(from|import) " -t py | sort | uniq -c | sort -rn | head -15
```

**2. 给 AI 的 Prompt 策略：**
> **Prompt:**
> 以下是项目的模块行数统计和核心依赖项。
> 1. **识别核心：** 哪些模块是当前的逻辑重心？
> 2. **预测瓶颈：** 考虑到项目使用了 [依赖库A, B]，如果用户量/数据量增加 10 倍，哪些部分最可能需要重构？
> 3. **生成 Roadmap：** 在 Roadmap 中加入“性能优化”和“模块解耦”的具体步骤。

---

### 维度三：功能对标分析 (Feature Gap Analysis)

这是最有杀伤力的方法：让 AI 扮演该领域的专家，将你的代码与行业标杆进行对比。

**1. 使用 Python 脚本提取核心 API/类：**
你可以运行之前在笔记中提到的 `RepoAnalyzer` 脚本，生成项目的“功能快照”。

**2. 给 AI 的 Prompt 策略：**
> **Prompt:**
> 这是一个 [项目类型，例如：向量数据库中间件] 项目。其核心类定义如下：
> [粘贴 RepoAnalyzer 的输出]
>
> 请参考该领域最成熟的项目（如 [Pinecone / Milvus]），回答：
> 1. **功能缺失：** 标杆项目有哪些关键特性（如：多租户隔离、元数据过滤、分布式共识）是本代码库尚未实现的？
> 2. **演进建议：** 请基于对比结果，为我生成一个从 V0.5 演进到 V2.0 的功能演进路线图。

---

### 总结：AI Agent 自动化操作流 (SOP)

如果你正在使用带有 Long Context 能力的 AI Agent（如 Claude 3.5 Sonnet 或 GPT-4o），直接执行以下指令：

1.  **输入指令：** "请对该代码仓进行 `static analysis`。先总结其目前的 `Core Value Proposition`。"
2.  **输入指令：** "对比 [[2025-12-10-高质量开源项目的特征]] 中的 10 个标志，列出差距。这构成 `Engineering Roadmap`。"
3.  **输入指令：** "分析现有 API 签名，对比行业内同类项目。这构成 `Feature Roadmap`。"
4.  **输入指令：** "最后，请合并这两部分，用 **Mermaid Gantt** 图表输出一份为期 3 个月的详细路线图。"

**关键点：** 即使没有 `TODO`，**代码的目录结构**和**依赖项 (requirements.txt)** 也会暴露出项目的成熟度和局限性。AI 对这些“元数据”的分析往往比单纯看注释更深刻。
