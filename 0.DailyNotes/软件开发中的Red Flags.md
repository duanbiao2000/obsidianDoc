# 软件开发中的 Red Flags（危险信号）

Red Flags 指那些**看似合理但实际上会导致问题**的想法或做法。


---

## 代码层面


| Red Flag | 为什么危险 |
|----------|-----------|
| "先让它能跑，以后再重构" | "以后"永远不会来 |
| "加个注释就行了" | 代码本身应该自解释 |
| "就改一行代码，不用测试" | 往往就是这一行出问题 |
| "这段代码没人动过，别碰它" | 技术债无限累积 |
| "复制粘贴比较快" | DRY 原则被破坏，维护噩梦 |


---

## 设计层面


| Red Flag | 为什么危险 |
|----------|-----------|
| "先把功能都做了，再考虑扩展性" | 架构腐烂，后期重写 |
| "我们需要一个通用框架" | 过度设计，YAGNI 原则 |
| "这个设计很巧妙" | 巧妙 = 复杂 = 难维护 |
| "以后肯定会用到" | 预测未来往往是错的 |


---

## 项目管理层面

| Red Flag | 为什么危险 |
|----------|-----------|
| "这个很简单，两天就能搞定" | 低估复杂度是常态 |
| "加人就能加快进度" | 布鲁克斯法则：人月神话 |

| "先上线再说" | 技术债 + 用户体验灾难 |
| "文档以后再写" | 永远不会写 |
| "这次我们不需要测试" | 必然出事 |


---

## 团队沟通层面

| Red Flag | 为什么危险 |
|----------|-----------|
| "我以为你知道..." | 假设是万恶之源 |
| "大家都懂的" | 知识诅咒 |
| "只有我能维护这块代码" | 单点故障，巴士因子 = 1 |
| "我们一直都是这样做的" | 拒绝改进 |

---

## 技术选型层面


| Red Flag | 为什么危险 |
|----------|-----------|
| "这个新框架很火" | 追热点，不考虑适用性 |
| "大厂都在用" | 你不是大厂 |
| "自己造轮子更可控" | 维护成本被严重低估 |
| "微服务能解决所有问题" | 分布式复杂度剧增 |

---


## 经典总结

> **"It works on my machine"**
> — 最危险的一句话

> **"We can fix it later"**
> — 第二危险的一句话

---

## 如何应对？

1. **质疑假设** — 问"为什么"
2. **小步迭代** — 避免大爆炸式发布
3. **代码审查** — 多人把关
4. **自动化测试** — 信任代码，不信任人
5. **写文档** — 现在就写
