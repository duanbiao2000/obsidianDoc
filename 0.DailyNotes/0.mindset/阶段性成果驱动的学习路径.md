## 0. 本质 (The Essence)

- **核心逻辑**：学习的目的不是“知道”，而是“做到”。
- **价值点**：通过强制产出（Deliverables）锚定知识，消除“看过就忘”的幻觉。
- **定位**：从知识囤积者进化为价值交付者。

---

## 1. 模式审计 (The Brutal Audit)

| 维度       | 传统积累 (Just-in-Case)                    | 成果驱动 (Just-in-Time)                      | 胜负手       |
| :------- | :------------------------------------- | :--------------------------------------- | :-------- |
| **逻辑顺序** | 理论 $\rightarrow$ 理论 $\rightarrow$ (忘了) | 成果定义 $\rightarrow$ 按需检索 $\rightarrow$ 交付 | 效率提升 300% |
| **认知负荷** | 试图搬走整座图书馆                              | 只取当前关卡所需的钥匙                              | 避免大脑“宕机”  |
| **记忆深度** | 被动输入 (幻觉掌握)                            | 主动提取 (提取练习效应)                            | 遗忘率 ↓80%  |
| **反馈周期** | 数月甚至数年                                 | 1-2 周一个迭代版本                              | 持续的多巴胺激励  |

## 2. 逻辑骨架 (The 3-Step Flow)

### **A. 定义成果 (Define)**

- **标准**：必须是**可演示 (Demo)**、**可部署 (Live)**、**可量化 (Metrics)** 的实体。
- **示例**：不是“学 React”，而是“在 Vercel 部署一个支持数据持久化的 Todo App”。

### **B. 逆向工程 (Reverse Engineering)**

- **公式**：`目标成果 -> 拆解组件 -> 识别最小必要知识 (20%) -> 忽略长尾细节 (80%)`。
- **逻辑**：不学整本手册，只学让齿轮转起来的那一章。

### **C. 迭代与复盘 (Iterate & Reflect)**

- **MVP (V1.0)**：两周内上线，能跑就行。
- **优化 (V2.0)**：引入高性能组件或缓存。
- **扩展 (V3.0)**：处理高并发或复杂交互。

---

## 3. 避坑指南 (Brutal Truths)

- **教程地狱 (Tutorial Hell)**：跟着敲 100 遍代码不叫学习，那是“脑力按摩”。丢掉教程，自己撞墙。
- **完美主义是阻碍**：在 MVP 阶段搞 CI/CD 和高覆盖率测试是纯粹的资源浪费。
- **刷题不等于工程力**：LeetCode 500 题救不了不会设计系统的你。算法要进场，而非进脑。
- **囤积癖必死**：买了没看的课、存了没读的 PDF 是思维的脂肪。

---

## 4. 决策树

1. **面对新领域感到迷茫？**
   - $\rightarrow$ 停止看书。先找一个最简单的 Demo 项目，强行模仿并跑通。
2. **觉得知识点太多学不完？**
   - $\rightarrow$ 执行 **80/20 法则**。问自己：“如果只学一个点就能让项目跑起来，是哪一个？”
3. **想评估学习效果？**
   - $\rightarrow$ 尝试把东西讲给 AI 听，或者发一个 GitHub Repo。收不到反馈的学习是无效的。

## 5. 极简学习清单模板

- [ ] **成果目标**：[Demo 链接 / GitHub Repo]
- [ ] **当前瓶颈**：[现在卡在哪一步？]
- [ ] **即时任务**：[为了解决瓶颈，只需搜索哪三个关键词？]
- [ ] **今日交付**：[晚上 10 点前能演示出的新功能]

---

**原则**：先做出能用的，再做出好用的，最后才是完美的。Just-in-Time 永远优于 Just-in-Case。
